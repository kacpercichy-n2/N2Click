1. **P2** — [src/components/TaskModal.tsx](/Users/kacpercichyn2/Documents/N2click/src/components/TaskModal.tsx:481)  
   **Issue:** Small positive estimates can persist as `0`. `0.1` passes the `estRaw > 0` check, then `snapHours(0.1)` becomes `0`, despite the comment/contract saying non-positive input clears to `null`. With the new budget enforcement, that creates a zero-budget task and blocks calendar inserts/growth unexpectedly.  
   **Suggested fix:** Snap first, then clear if the snapped value is `<= 0`:
   ```ts
   const snapped = snapHours(estRaw);
   const estimatedHours = invalid || snapped <= 0 ? null : snapped;
   ```
   Use the same normalized value for the live estimate comparison.

2. **P3** — [src/components/TaskModal.tsx](/Users/kacpercichyn2/Documents/N2click/src/components/TaskModal.tsx:519)  
   **Issue:** The over-budget banner compares against raw `estimatedRaw`, while save persists the snapped estimate. This can show false warnings, e.g. `1.13h` displays as over budget against `1.25h` planned, but saves as `1.25h`.  
   **Suggested fix:** Derive one `normalizedEstimate` value and use it for save, display, and `overBudget`.

3. **P3** — [src/components/WeekView.tsx](/Users/kacpercichyn2/Documents/N2click/src/components/WeekView.tsx:704)  
   **Issue:** The insert form checks raw `hoursRaw` against allowance before reducer-equivalent snapping/clamping. Manual values like `1.01` can be blocked even though the reducer would snap to `1.0` and accept it.  
   **Suggested fix:** Compute `snappedInsertHours = snapHours(Math.min(24, parsedHours))` and use that for `overAllowance`, `confirmInsert`, and the disabled state.
