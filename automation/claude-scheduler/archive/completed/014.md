Release-hardening bundle 2: stop TaskModal saves from deleting valid same-day calendar blocks.

Context: the audit reproduced two blocks for the same task/person/day being collapsed to one when the task is saved unchanged. Calendar workflows intentionally support multiple blocks, so this is silent data loss and a release blocker.

Scope:
- Trace the TaskModal draft/allocation representation and SAVE_TASK rebuild path end to end before editing.
- Redesign the edit model so existing dated and bin workload entries are preserved by entry identity, planned hours, ordering, and startMinutes unless the user explicitly changes them.
- An unchanged Save must be structurally lossless. An intentional allocation edit must have deterministic semantics that never silently deletes another same-day block.
- Preserve existing collision, estimate, assignment, bin, and sort-index invariants. Do not solve this by disabling multiple same-day blocks.
- Add reducer tests for two or more blocks on one person/day, mixed dated/bin entries, unchanged save, explicit changes, and reload persistence.
- Reproduce the reported browser case and add it to the browser regression harness if practical.

Acceptance:
- Saving a task without schedule edits preserves every existing workload row and timing exactly.
- Any destructive scheduling action is explicit in UI and covered by tests.
- No regression to bin drag/drop, split, resize, assignment, or budget behavior.
- Run fresh direct tsc, vitest, production build, Chromium and WebKit checks.
- Update docs/handoff notes with the workload-entry identity invariant.
