# Protect dirty task and project navigation

## Risk and routing
- Risk: medium
- Route: architect → developer → reviewer
- Codex review: conditional — run only if the diff crosses an undeclared persistence boundary or reviewer uncertainty remains

## Wiki context
- `openwiki/n2hub/ui-navigation-and-onboarding.md`
- `openwiki/n2hub/testing-and-automation.md`

## Expected touchpoints
- `src/App.tsx`
- `src/components/TaskModal.tsx`
- `src/pages/ProjectDetailPage.tsx`
- `src/utils/dirtyRegistry.ts`
- `src/utils/dirtyRegistry.test.ts`

## Invariants
- Cancelled navigation retains the dirty edit.
- Clean navigation stays immediate and existing close-button confirmations remain intact.
- Unrelated routes and forms do not receive a global blocker.

## Scope
- Use the shared dirty-edit boundary for TaskModal and ProjectDetailPage.
- Protect browser Back/Forward, route changes and application links with clear Polish confirmation.
- Add focused cancel/confirm coverage.

## Out of scope
- Reducer validation, activity attribution, authentication, permissions and persistence-conflict behavior.
- Dirty guards for unrelated pages.

## Acceptance
- Dirty task/project edits cannot be silently discarded by browser or in-app navigation.
- Cancel retains the edit; confirm proceeds; clean navigation is unchanged.

## Verification
- Worker: run `npx vitest run src/utils/dirtyRegistry.test.ts` plus the closest affected focused tests.
- Browser: run only the focused dirty-navigation scenario in Chromium; add WebKit only if browser-specific behavior differs.
- Scheduler: owns the single final `npm test && npm run build` gate.
