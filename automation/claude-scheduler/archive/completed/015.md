Release-hardening bundle 3: make task/project status semantics stable and safe to administer.

Context: completion is currently inferred from the final active status. Reordering or archiving statuses changes historical meaning, can resurrect overdue tasks, and can hide used projects from Kanban. This is high severity.

Scope:
- Add an explicit stable semantic property for completion, such as isDone, to the status model with safe migration defaults that preserve current intended done statuses.
- All selectors for overdue, agenda, my-work, dashboard, and related task logic must use the explicit semantic rather than pipeline position.
- Status admin must not make referenced projects/tasks disappear. Either keep a visible archived/orphan column with clear labeling or prevent unsafe archival and require reassignment; choose the least surprising existing-pattern fit.
- Protect core pipeline invariants: users must not archive/delete all active statuses or remove the only completion state without a clear migration/reassignment path.
- Keep operational status separate from future formal/sales status; do not add project-card fields in this hardening run.

Acceptance:
- Reorder does not alter which tasks are completed.
- Archiving a used status neither hides records nor revives completed work as overdue.
- Polish UI explains any blocked/archive/reassignment action.
- Add migration, selector, reducer, and browser tests for the reproduced scenarios.
- Run fresh direct tsc, vitest, production build, Chromium and WebKit checks; update docs/handoff notes.
