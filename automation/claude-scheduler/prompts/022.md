Production-readiness bundle 10: define the backend migration boundary without pretending the frontend is secure.

## Wiki context

- `openwiki/n2hub/state-and-persistence.md`
- `openwiki/n2hub/ui-navigation-and-onboarding.md`

## Expected touchpoints

- `src/store/storage.ts`, `src/store/AppStore.tsx`, `src/types.ts`, `docs/decisions/`, `docs/plans/`

Context: all identities, permissions, passwords, sessions, and business data currently live in mutable localStorage. This is the primary production blocker. A scheduled unattended run must not choose a cloud provider, create accounts, deploy a database, or migrate real data without explicit human approval.

Scope:
- Produce a concrete, repo-local architecture and migration plan for replacing localStorage with a backend: data model, authentication/session boundary, server-side authorization, password handling/recovery, audit trail, revisions/conflicts, backups/export, and rollout/rollback.
- Identify exact frontend seams that must become repositories/API adapters, plus a staged migration that keeps the prototype usable while a backend is introduced.
- Add only low-risk preparatory code that is provider-neutral and fully tested, such as typed repository interfaces or a storage adapter boundary. Do not introduce credentials, external SDKs, hidden network calls, or mock security claims.
- Add a production security baseline checklist for CSP/security headers, error reporting, audit retention, dependency/secret handling, and data privacy. Apply a header only if it is tested not to break the existing app; otherwise document it as a decision for the human.
- Preserve every current local workflow and make the prototype limitation explicit in UI/docs where appropriate.

Acceptance:
- A human can choose a backend and execute the migration from a concrete staged plan without rediscovering the current architecture.
- No external service, credential, or real-data migration occurs.
- Any adapter boundary is small, provider-neutral, and covered by tests.
- Run fresh direct tsc, vitest, production build, and browser smoke checks; update docs/handoff notes.
