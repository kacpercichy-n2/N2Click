# Define the provider-neutral backend migration boundary

## Risk and routing
- Risk: high
- Route: architect → reviewer
- Codex review: required

## Wiki context
- `openwiki/n2hub/state-and-persistence.md`
- `openwiki/n2hub/ui-navigation-and-onboarding.md`

## Expected touchpoints
- `src/store/storage.ts`
- `src/store/AppStore.tsx`
- `src/store/permissions.ts`
- `src/types.ts`
- `new: docs/decisions/backend-migration-boundary.md`
- `new: docs/plans/backend-migration-plan.md`

## Invariants
- No provider, external service, credential, deployment or real-data migration is selected or created.
- Current local workflows remain unchanged.
- The plan never describes frontend checks/localStorage as a security boundary.

## Scope
- Produce decision and staged migration documents covering data model, sessions, server authorization, password recovery, audit, revisions/conflicts, backup/export and rollout/rollback.
- Identify exact frontend repository/API seams and explicit human decisions required before implementation.
- Add a production security checklist for headers, errors, audit retention, dependencies, secrets and privacy.

## Out of scope
- Production-code adapters, SDKs, headers, UI changes, credentials and browser checks.
- Choosing a backend provider or starting migration work.

## Acceptance
- A human can compare provider options and approve a later bounded implementation package without rediscovering the frontend boundaries.
- Every implementation stage has prerequisites, rollback and validation criteria.
- No code or runtime behavior changes.

## Verification
- Architect: cross-check named seams read-only and write only the decision/plan deliverables.
- Browser: none — documentation/architecture task with no runtime changes.
- Scheduler: owns the single final `npm test && npm run build` gate.
