# Validate persistent reducer commands

## Risk and routing
- Risk: high
- Route: architect → developer → reviewer
- Codex review: required

## Wiki context
- `openwiki/n2hub/state-and-persistence.md`
- `openwiki/n2hub/testing-and-automation.md`

## Expected touchpoints
- `src/store/AppStore.tsx`
- `src/types.ts`
- `src/store/*.test.ts`

## Invariants
- Rejected commands preserve the previous state reference.
- Valid `SAVE_TASK` reconciliation, bin identity, date guards and status semantics remain unchanged.
- Persisted dates and required entity references remain valid.

## Scope
- Add one shared reducer-command validation boundary for stale IDs, required titles, dates, estimates and status/project references.
- Cover task, project, milestone, status, person and current-user mutations only where they share that boundary.
- Add focused reducer tests for rejected and valid legacy payloads.

## Out of scope
- Navigation guards, activity attribution, authentication, permissions, storage conflicts and browser routing.
- Broad cleanup of unrelated reducer actions.

## Acceptance
- Invalid or stale payloads cannot persist malformed entities and return the original state reference.
- Valid legacy edits and existing workload/status behavior remain green.
- The diff stays inside the declared reducer boundary and its focused tests.

## Verification
- Worker: run the smallest relevant Vitest files while iterating.
- Browser: none — reducer behavior is covered by focused unit tests.
- Scheduler: owns the single final `npm test && npm run build` gate.
